## 1. 产品概述

### 1.1 产品名称

**PVE MCP Server** — 基于 MCP 协议的 Proxmox VE 智能运维助手

### 1.2 产品定义

PVE MCP Server 是一个实现了 Model Context Protocol (MCP) 的服务端，它将 Proxmox VE 虚拟化平台的 REST API 能力暴露为标准化的工具（Tools）和资源（Resources），使 AI 助手（如 Claude）能够直接查询、分析和管理 PVE 集群资源。

### 1.3 产品愿景

让虚拟化运维从"登录 Web 控制台逐项查看"进化为"用自然语言对话获取洞察"，大幅提升运维效率和问题排查速度。

## 2. 背景与动机

### 2.1 现状痛点

用户在日常运维 PVE 时面临以下核心问题：

| # | 痛点 | 典型场景 | 影响 |
|---|------|----------|------|
| 1 | **负载感知滞后** | PVE 负载突然飙高，不知道是哪台 VM 造成的 | 问题排查慢，业务影响时间长 |
| 2 | **资源全貌模糊** | 不清楚集群整体资源使用率，无法快速判断是否还有扩容空间 | 决策靠经验，缺乏数据支撑 |
| 3 | **VM 信息碎片化** | 想了解某台 VM 的配置需要点进多个页面 | 信息收集效率低 |
| 4 | **资源分配不透明** | 不知道哪些 VM 消耗资源最多，分配是否合理 | 资源浪费或分配不均 |
| 5 | **历史趋势难以获取** | 想看过去一周的负载趋势需要手动截图 | 无法做容量规划 |

### 1.2 为什么选择 MCP

| 方案 | 优点 | 缺点 |
|------|------|------|
| 传统 Web UI | 功能全面 | 需要人工操作，无法自动化 |
| 脚本 + CLI | 可自动化 | 需要编程能力，维护成本高 |
| Grafana + Prometheus | 可视化好 | 部署复杂，查询语言有学习成本 |
| **MCP + AI 助手** | 自然语言交互，智能分析 | 首次建设成本 |

**MCP 方案的独特价值**：
- 自然语言交互：不记命令、不写 PromQL
- 智能上下文：AI 能关联多维度信息给出洞察
- 即时可用：无需部署额外监控栈
- 可扩展：未来可接入更多工具能力

## 3. 目标用户

### 3.1 用户画像

| 角色 | 特征 | 核心诉求 |
|------|------|----------|
| **Homelab 玩家** | 运行 1-3 台 PVE 节点，托管个人服务 | 快速了解资源状态，排查问题 |
| **中小企业运维** | 管理 3-10 台 PVE 集群 | 高效监控，减少登录 Web 控制台的频率 |
| **开发者** | 在 PVE 上运行开发/测试环境 | 快速查看 VM 状态，管理开发环境 |

### 3.2 用户规模预期

- **Phase 1 (MVP)**: 单节点用户（Homelab 场景）
- **Phase 2**: 多节点集群用户（中小企业场景）

## 4. 产品目标

### 4.1 核心目标

| # | 目标 | 衡量指标 | 目标值 |
|---|------|----------|--------|
| G1 | 快速了解节点负载 | 获取节点状态的响应时间 | < 2s |
| G2 | 清晰掌握资源分配 | 能回答"哪台 VM 消耗资源最多" | 支持 Top N 查询 |
| G3 | VM 信息一目了然 | 能回答"某台 VM 的配置和状态" | 支持单 VM 详情查询 |
| G4 | 资源使用可追溯 | 能回答"过去一周负载趋势" | 支持 RRD 数据查询 |

### 4.2 非目标（明确不做）

- ❌ 替代 PVE Web UI（不重新实现完整管理界面）
- ❌ 实时告警系统（建议配合 Prometheus/Grafana）
- ❌ 自动扩缩容（超出 MCP 服务范围）
- ❌ 备份管理（复杂度高，优先级低）

## 6. 用户场景 (User Stories)

### 6.1 场景 1: 负载飙升排查

**场景描述**：
用户收到告警"CPU 使用率超过 80%"，需要快速定位原因。

**用户流程**：
```
用户: "PVE 负载很高，帮我看看怎么回事"

AI 助手调用:
1. get_node_status() → 获取节点状态，确认 CPU 85%
2. list_vms(node="pve-node01") → 获取 VM 列表
3. 按 CPU 排序，发现 VM 101 (db-mysql) 占用 78%

AI 回复:
"节点 pve-node01 当前 CPU 使用率 85%，主要由 VM 101 (db-mysql) 贡献。
该 VM CPU 使用率 78%，内存使用率 97.5%。
建议检查 MySQL 是否有慢查询或连接数过多。"
```

### 6.2 场景 2: 容量规划

**场景描述**：
用户想新增一台 VM，需要评估当前资源是否充足。

**用户流程**：
```
用户: "我想新建一个 4 核 8G 的 VM，现在资源够吗？"

AI 助手调用:
1. get_resource_overview() → 获取资源概览
2. 发现 CPU 超分配 3x，内存剩余 4.8GB

AI 回复:
"⚠️ 资源紧张，不建议创建:
- CPU: 已超分配 3x，新建后将达 3.5x
- 内存: 仅剩 4.8 GB，不满足 8 GB 需求

建议:
1. 停止 3 台已关机的 VM 释放资源
2. 考虑升级物理内存至 64 GB
3. 或减少新 VM 配置为 2 核 4 GB"
```

### 6.3 场景 3: 日常巡检

**场景描述**：
用户想快速了解整体健康状况。

**用户流程**：
```
用户: "帮我做个 PVE 健康检查"

AI 助手调用:
1. get_node_status() → 节点状态
2. list_vms() → VM 列表
3. get_resource_overview() → 资源概览

AI 回复:
"📊 PVE 健康检查报告

✅ 整体状态: 良好

🖥️ 节点: pve-node01
   - 运行 15 天，无异常
   - CPU 45%，内存 57%，负载适中

💻 虚拟机: 8 台 (5 运行，3 停止)
   - ⚠️ VM 101 内存使用率 97.5%，需关注
   - ✅ 其余 VM 状态正常

💾 存储: 使用 42.8%，空间充足

📈 趋势: CPU 缓慢上升，建议持续关注"
```

## 7. 技术约束

### 7.1 环境要求

| 项目 | 要求 |
|------|------|
| Python 版本 | >= 3.14 |
| PVE 版本 | >= 7.0 (API v2) |
| 网络 | MCP Server 需能访问 PVE 节点 8006 端口 |
| 认证 | 需创建 PVE API Token 或使用用户密码 |

### 7.2 API 限制

| 限制项 | 说明 |
|--------|------|
| 请求频率 | PVE API 无官方限流，但建议 < 10 req/s |
| 超时 | 建议单次请求超时 10-30 秒 |
| 并发 | 避免对同一节点并发执行写操作 |

### 7.3 安全要求

| # | 要求 |
|---|------|
| S1 | API Token 存储在环境变量或安全配置文件中 |
| S2 | Token 权限遵循最小权限原则 |
| S3 | 敏感信息（密码、Token）不出现在日志中 |
| S4 | 只读操作优先，写操作需要显式确认 |
